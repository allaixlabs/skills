# DOD.md — 3단계 다차원 완료 기준 상세

AI는 "다 됐어?"라는 질문에 관대하게 "완료"라 답하는 경향이 있다. 이를 막기 위해
통과해야 할 단계를 **세 가지 유형**으로 쪼개어 시스템적으로 검증 루프를 돌린다.

---

## ① 필수 통과 기준 (Pass/Fail Gate)

> 80점·90점 같은 타협이 없다. **단 하나라도 실패하면 다음 루프로 진행 불가 → 롤백.**

| 영역 | 기준 | 검증 |
|------|------|------|
| 빌드/컴파일 | 소스 빌드 + 타입체크(tsc 등) 성공 | exit 0, 에러 0 |
| 코드 스타일 | 정적 분석(Lint) 에러 제로 | 에러 0 |
| 기능 무결성 | 유닛·통합 테스트 100% 성공 | 실패 0 |
| 인프라/보안 | DB 마이그레이션 정상 + 시크릿(API Key 등) 노출 차단 | 마이그레이션 성공, 노출 0 |

---

## ② 정량 측정 기준 (Metrics Data)

> 데이터·숫자로 파악. 주기적으로 실행해 대시보드/리포트로 도출.

- **테스트 커버리지**: 핵심 로직 커버리지 하한선 충족 여부
- **성능/비용**: API 응답 속도, 프론트엔드 번들 크기 변화 추이
- **안정성 지표**: 모니터링상 에러 로그 발생률, API 호출 실패율
- **비즈니스 특화 테스트**: 도메인별 예외 케이스 자동 검증 비율
  - 예) 일정 관리 앱: 날짜 계산 유틸 예외 처리, 반복 일정 생성/수정/삭제 엣지케이스

---

## ③ 정성 평가 기준 (Independent Verifier > Self-Criticism)

> 모델은 자기 출력 자기비판에 약하다. **독립 컨텍스트 채점이 1순위**, 자기평가는 차선책이다.

**독립 검증자 절차 (권장):**
1. fresh-context 서브에이전트를 띄운다 (작업하던 컨텍스트와 분리된 새 창).
2. 입력은 딱 세 가지만: `loop.md` + `git diff` + ①② 실행 로그. **작업 경위·중간 대화·자기 해명은 주지 않는다** — 채점자가 작업자의 서사에 오염되지 않게.
3. 산출: 항목별 1~5점 + 감점 근거 + 액션. 작업자는 이 결과를 수정 없이 리포트에 싣는다(반박은 다음 루프에서 수정으로 증명).
4. 서브에이전트를 띄울 수 없는 환경(단독 비대화형 등)에서만 아래 자기평가로 대체하고, 리포트에 "자기평가(사유)"를 명시한다.

**평가 항목:**
- 아키텍처 패턴의 적절성
- 가독성 및 유지보수 가능성
- 예외 처리 및 에러 메시지의 친절도

**방어 장치 (근거·액션 강제):**
1~5점 스케일로 평가하되, 감점(5점 미만) 요소에는 **반드시**
구체적 **근거(Reason)** 와 다음 루프에서 수행할 **수정 액션(Action Item)** 을
아래 포맷으로 제출해야만 완료로 인정한다.

**예시 리포트 포맷:**
```
[항목: 유저플로우] [점수: 3점]
[근거: 일정 수정 후 캘린더 화면 상태가 실시간 동기화되는지 검증하는 자동화 테스트가 누락됨]
[액션: 해당 상태 갱신 검증 테스트 코드를 추가 구현할 것]
```

```
[항목: 예외 처리] [점수: 4점]
[근거: 네트워크 타임아웃 시 사용자 메시지가 'Error'로만 표시되어 불친절]
[액션: 재시도 안내 문구와 함께 사용자 친화적 에러 메시지로 교체]
```

## 루프 운영 원칙

1. ① → ② → ③ 순서로 검증. 시작 전 `.loop/lessons.md`를 먼저 읽는다(consult — 같은 원인 재유도 금지).
2. ① 게이트 실패 시 ②③ 진입 금지. 즉시 롤백·재작업.
3. **실패는 증류한다**: FAIL마다 원인을 검증한 뒤 `.loop/lessons.md`에
   `[날짜] [게이트] 증상 → 검증된 원인 → 일반 규칙` 1줄을 남긴다. 검증 안 된 추측("아마 X?")은 기록 금지 —
   fail → investigate → verify → distill → (다음 루프가) consult.
4. **동일 게이트 연속 3회 실패 시 루프 중단** — 남은 항목과 검증된 진단을 사용자에게 보고한다.
5. ③ 정성 평가의 액션 아이템은 **다음 루프의 입력**이 된다.
6. 완료 선언은 ①②③ 통과 + 리포트 출력이 모두 끝난 뒤에만 가능. 리포트에 검증 주체·라운드 수를 기록한다.
